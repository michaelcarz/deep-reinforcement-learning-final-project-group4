**課程：** 深度強化學習  
**系統環境：** Windows (RTX 4070 Ti CUDA) / macOS (Apple Silicon MPS)  
**模擬平台：** MuJoCo Humanoid-v4 + Gymnasium + Stable-Baselines3

## 一、摘要

> **一句話概括：一隻雙足機器人，在走路走到一半時突然壞了一條腿，它沒有被重新編程，卻自己學會了跛著走下去。**

本研究探討雙足機器人在關節突發失效（電機卡死）時，能否透過 PPO 強化學習自主學會代償行走。我們沒有寫任何「如果右膝壞了就用左腿支撐」的規則，而是只做了三件事：

1. **告訴機器人哪個關節壞了**（Health Vector，17 維健康向量）
2. **讓壞掉的關節真的動不了**（Action Masking Gate，物理遮蔽）
3. **只要帶著傷還活著，就給額外獎勵**（Compensation Bonus +3.0）

然後機器人自己摸索出了怎麼用剩下的健康關節撐住、扭動、拖行——這就是所謂的「自主湧現 (Emergent)」代償步態。

**最終成果：** 機器人在關節動態鎖死後仍能存活 67.7%（V1）/ 40%（V2 平滑版）的步數，成功展現自主代償行為。

## 二、研究動機

工業用雙足機器人在實際運作中，關節電機可能因磨損、過熱或意外碰撞而突然卡死。傳統 PID/MPC 控制器在系統結構突然改變時無法即時應對，機器人會立刻摔倒。

**我們想回答的問題：**

> 雙足機器人能否透過深度強化學習（PPO），在完全沒有人工設計代償策略的前提下，面對突發性關節硬體鎖死，自主湧現出補償性的跛行步態？

**具體目標：**
1. 讓機器人先學會穩定走路（健康步態 ≥ 900 步 / 1000 步）
2. 讓機器人在關節壞掉後，不用重新訓練就能自己調整走法繼續前進

## 四、系統架構

### 4.1 觀測空間擴充（376 維 → 410 維）

在 Humanoid-v4 原本的 376 維觀測基礎上，額外加入兩組向量：

| 觀測分量 | 維度 | 白話解釋 |
|---------|:---:|---------|
| 原始觀測（位置、速度、力矩等） | 376 | 機器人「看到」自己身體的狀態 |
| **健康向量** `health_vector` | 17 | 機器人「感覺」到每個關節有沒有壞（1.0=正常, 0.5=半損, 0.0=鎖死） |
| **上一步動作** `prev_action` | 17 | 機器人「記得」自己上一步做了什麼動作 |

程式碼：

```python
# cybernetic_resilience_smooth.py

def _get_obs(self, obs):
    """把健康向量和上一步動作接在原始觀測後面，讓機器人能感知傷勢"""
    return np.concatenate([obs, self.health_vector, self.prev_action])
```

### 4.2 動作遮蔽閘（Action Masking Gate）

壞掉的關節不是「叫它不要動」，而是在物理層面直接把控制信號歸零。就像真實的電機卡死一樣——不管大腦發什麼指令，那個關節就是不會動。

```python
# 策略網路輸出的動作 × 健康向量 = 實際執行的動作
# 健康值 = 0.0 的關節，不管網路輸出什麼，乘上去都是 0
masked_action = action * self.health_vector
obs, reward, terminated, truncated, info = self.env.step(masked_action)
```

### 4.3 漸進式關節退化

關節不是瞬間壞掉的。就像真實的機械磨損一樣，先是「出力變弱」，最後才「完全鎖死」：

```python
if self.step_count >= self.shock_step:
    steps_since_shock = self.step_count - self.shock_step
    if steps_since_shock < 50:
        current_health = 0.5   # 前 50 步：出力減半（半損狀態）
    else:
        current_health = 0.0   # 50 步後：完全鎖死（不能動）
    for j in self.locked_joints_this_episode:
        self.health_vector[j] = current_health
```

### 4.4 獎勵函數設計

獎勵函數就是「教機器人什麼是好、什麼是壞」的規則：

```python
# 1. 速度上限：走太快扣分（不准衝刺，穩穩走就好）
if forward_reward > 1.875:
    reward -= (forward_reward - 1.875)

# 2. 動作平滑懲罰：兩步之間動作差太多扣分（防止關節抖動）
action_diff = action - self.prev_action
smoothing_penalty = 0.05 * np.sum(np.square(action_diff))
reward -= smoothing_penalty

# 3. 軀幹穩定懲罰：身體左右晃、上下跳都扣分
qvel = self.env.unwrapped.data.qvel
torso_penalty = 0.5 * (qvel[1]**2 + qvel[2]**2) + 0.1 * np.sum(np.square(qvel[3:6]))
reward -= torso_penalty

# 4. 代償生存獎勵：帶著傷還活著？每步加 3 分！（鼓勵它撐下去）
if any(h < 1.0 for h in self.health_vector) and not terminated:
    reward += 3.0
```

## 五、訓練方法

### 5.1 兩階段課程學習

整個訓練分成兩大階段——就像教小孩「先學走路，再學帶傷走路」：

```
Phase 1（先學走路）                Phase 2（再學帶傷走路）
┌──────────────────────┐         ┌──────────────────────────────┐
│ 全部關節健康            │         │ Stage 2: 起步就鎖死 1~2 個關節  │
│ 高度限制 z ≥ 1.0m      │  ──→   │ Stage 3: 走到一半突然鎖死       │
│ 不准蹲、不准爬、只能站著走│         │ 高度放寬 z ≥ 0.5m（允許歪斜）    │
│ 35M 步（V1）/ 5M 步（V2）│         │ 65M 步（V1）/ 10M 步（V2）     │
└──────────────────────┘         └──────────────────────────────┘
```

**為什麼不能直接從頭訓練帶傷走路？**

因為機器人會作弊——它會發現「跪在地上不動」就不會摔倒，反而能得到獎勵。所以必須先用嚴格的高度限制（z ≥ 1.0m）逼它學會站著走路，確認真的會走了，再把腿打斷看它怎麼辦。

### 5.2 PPO 超參數設定

```python
model = PPO(
    policy="MlpPolicy",
    n_steps=512,           # 每走 512 步就更新一次策略
    batch_size=64,
    n_epochs=10,
    learning_rate=linear_schedule(2.5e-4, 2.5e-5),  # 學習率從大到小慢慢降
    clip_range=0.2,
    target_kl=0.05,        # 最關鍵！更新時策略變化太大就立刻停（防崩）
    use_sde=False,         # 不用 gSDE（會導致噪音爆炸，詳見第六章）
    policy_kwargs=dict(net_arch=[256, 256]),  # 兩層各 256 個神經元
)
```

### 5.3 為什麼選這些超參數？

| 參數 | 設定值 | 白話解釋 |
|------|--------|---------|
| `target_kl=0.05` | 0.05 | 「如果這次更新讓策略變化超過 5%，立刻停」。沒有這個的話，策略一次更新太大就會崩潰，再也恢復不了 |
| `use_sde=False` | 關閉 | gSDE 是一種探索方式，但我們實驗發現它的噪音會從正常的 1.0 膨脹到 11+，直接把策略信號淹沒 |
| `net_arch=[256,256]` | 兩層 256 | 太大（如 512×512）在關節鎖死的劇烈環境變化下容易梯度爆炸 |
| `n_steps=512` | 512 | 比預設 2048 小，更頻繁地更新，搭配 target_kl 效果更好 |

## 六、實驗歷程 — 16 次失敗到成功的過程

這個專案不是一次就成功的。它經歷了 16 次實驗，每次失敗都帶來關鍵的診斷與修正。這個「失敗→診斷→突破」的過程本身就是最有價值的學習經驗。

### 🔴 第一幕：天真的嘗試（Run 1-2）

直接在有缺陷的環境裡訓練 → 機器人摔了就摔了，毫無代償能力。

**教訓：** 不能直接在「壞的環境」裡從零訓練。

### 🟡 第二幕：跪地作弊（Run 3-5）

機器人發現「跪在地上不動」就不會被判定摔倒 → 獎勵駭客 (Reward Hacking)。

**教訓：** 必須先用嚴格的高度限制逼它站著走 → Z 軸隔離的必要性。

### 🟡 第三幕：探索噪音失控（Run 6-14）

gSDE 探索機制的標準差從 1.0 膨脹到 11.1 → 機器人的動作完全被隨機噪音淹沒。某些 Run 初始 entropy 異常（-67 vs 正常 -24）→ 從崩壞的舊 checkpoint 啟動。

**教訓：** gSDE 在超長訓練中是定時炸彈，必須換掉。

### 🟢 第四幕：關鍵突破（Run 15）

三個修復同時上線：

| 修復項目 | 做了什麼 | 效果 |
|---------|---------|------|
| 停用 gSDE | 改用標準高斯噪音 | std 全程穩定在 1.0~1.2 |
| 加入 `target_kl=0.05` | KL 散度超標就停止更新 | 策略不再崩潰 |
| `n_steps` 2048→512 | 更頻繁但更小的更新 | 學習更穩定 |

**結果：** 首次實現穩定收斂到 976 步！但因伺服器重啟中斷...

### 🏆 第五幕：最終完勝（Run 16）

用 Run 15 的配置完整跑完 100M 步：

| 訓練階段 | 步數 | 平均存活步數 | 發生了什麼 |
|---------|------|:-----------:|----------|
| Phase 1 起步 | 3M | ~480 | 開始學走路 |
| Phase 1 中期 | 10M | ~850 | 走得越來越穩 |
| Phase 1 完成 | 35M | **957** | 近乎完美的步態 |
| Phase 2 切入 | +0M | 861→**225** | 關節壞了！存活暴跌（正常） |
| Phase 2 適應 | +3.6M | 322 | 開始學會代償了 |
| Phase 2 完成 | +65M | **637~651** | 穩定的代償行走 |

**為什麼 V2 韌性數字比 V1 低，但不是退步？**

V2 多了「不准抖動」「不准亂晃」的約束。這等於告訴機器人：光是活下來還不夠，你得走得好看。所以 V2 其實是在探討一個工程取捨問題——**平滑度 (Smoothness) vs 韌性 (Resilience)**：追求更穩定自然的步態，必然要犧牲一些極端情況下的存活能力。

### 8.3 韌性指數計算公式

$$I_{resilience} = \frac{R_{static} + R_{dynamic}}{2}$$

其中：

$$R_{static} = \frac{\bar{T}_{static}}{\bar{T}_{healthy}} \times 100\%$$

$$R_{dynamic} = \frac{\bar{T}_{dynamic}}{\bar{T}_{healthy}} \times 100\%$$

## 九、三大核心發現

### 發現一：環境隔離是課程學習的必要條件

> 如果一開始就放寬高度限制，機器人會找到「跪地前行」的作弊策略，永遠學不會站著走。

就像教小孩先學走路再學跑步——不能一開始就在泥地裡讓他跑。必須先用嚴格限制（z ≥ 1.0m）訓練正常步態，確認會走了，才能放寬限制（z ≥ 0.5m）訓練代償。

### 發現二：膝關節鎖死是致命的，髖關節鎖死是可以代償的

靜態鎖死測試中：
- 鎖住**膝關節**（index 6, 10）→ 幾乎必摔，因為失去了整條腿的支撐結構
- 鎖住**髖關節**（index 4, 8）→ 可以走滿 1000 步，靠對側骨盆擺動代償

**白話解釋：** 膝關節像是腿的「支柱」，壞了等於整條腿軟掉；髖關節只控制腿的擺動方向，壞了還可以用另一側的髖來補。

### 發現三：走路途中壞掉，比一開始就壞掉更容易存活

> 動態韌性 67.7% >> 靜態韌性 38.8%

**為什麼？** 三個原因：
1. **慣性**：走路時身體有前向動量，提供物理上的恢復力
2. **緩衝時間**：漸進式退化給了 50 步的過渡期，機器人有時間感知 health_vector 的變化並調整策略
3. **零動量困境**：靜態失效是從完全靜止開始，代償難度天然更高
